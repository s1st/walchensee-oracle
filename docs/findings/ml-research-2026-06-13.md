# ML methodology for the Walchensee thermal-wind classifier — research report

**Phase 0, Parts 4 + 5.** This document is the foundation for the Phase C "ceiling spike" that will train a `HistGradientBoostingClassifier` (and competitors) on ~1,900 in-season ground-truth days and benchmark them against the 14-rule heuristic baseline. It is a research review, not a design document; the implementation plan follows after the spike's empirical results are in.

## TL;DR

Five things the developer of Phase C needs to internalise before writing any code:

- **Trees, not nets — but include TabPFN.** At 1–2k rows × 15–20 features, gradient-boosted decision trees (GBDT) remain the strongest single model family (Grinsztajn et al. 2022, Shwartz-Ziv & Armon 2022). TabPFN-style in-context-learning transformers (Hollmann et al. 2023, McElfresh et al. 2023) are an active threat that has to be benchmarked at this sample size; deep tabular nets (TabNet, FT-Transformer) lose to GBDT here.
- **`HistGradientBoostingClassifier` is the right sklearn entry point** — it handles the project's two block-missing features natively (NaN) and supports `class_weight='balanced'` plus per-class dicts. The v1.2+ `class_weight` argument is essential. Caveat: the 10x speedup over `GradientBoostingClassifier` only kicks in at `n_samples ≥ 10,000`; for the project's regime, **`min_samples_leaf` tuning matters more than training time**, and the speed gain is irrelevant.
- **Validate temporally, block by year.** `TimeSeriesSplit` plus a year-blocked holdout (train ≤ 2022, test ≥ 2023) is the only honest evaluation on autocorrelated daily weather. Roberts et al. (2017) gives the formal justification; sklearn's `common_pitfalls` page shows what leakage looks like in CV scores.
- **Score with Peirce + RPS + relative economic value — never accuracy.** The current rule baseline's thermal-label Peirce skill score is +0.107 (barely above chance). Accuracy is dominated by the majority class; the Heidke Skill Score double-counts the easy cases; only Peirce (HK/TSS) and the Brier/RPS families are proper skill scores that zero out constant forecasts.
- **Use McNemar for fixed test sets, 5×2cv paired t for resampled evaluation.** Dietterich (1998) is still the cleanest reference. For the project, that means: a year-blocked holdout (2023+ as test) → McNemar's exact binomial if any disagreement cell is < 25, else χ²; and for hyperparameter sweeps → 5x2cv paired t. Treat TabPFN as a separate family and re-run the appropriate test.

A separate but related point: **temperature scaling (now native in `CalibratedClassifierCV` v1.8+) should be the default multiclass calibration method**, not sigmoid-OvR, because the OvR map can cross class-assignment boundaries and silently change the predicted class.

## 1. Introduction & scope

This report covers **two methodology areas** of the Phase 0 brief that are not yet documented in-repo:

- **Part 4 — small-n tabular ML methodology.** Which model families to consider for ~1,900 in-season days and 15–20 features; how to calibrate probabilities for a 3-class ordinal target; whether to use ordinal regression or cost-sensitive multinomial classification; how to handle class imbalance; how to do leakage-aware temporal validation; how to avoid overfit; and how to handle features that are block-missing for a whole historical era.

The thermal-wind driver, the thermik-delta pressure signal, the foehn-delta override, the Föhn detection, the slope-heating trigger, the spatial progression along the lake, and the rationale for the current 15-feature rule set are already documented in `docs/thermal-model.md` and `docs/future-factors.md` and are taken as given here. The reader is expected to be familiar with the project; this report sits downstream of those.

The **current rule baseline** is the 14-rule heuristic in `src/oracle/knowledge/rules.py`, which on the 3,331-day replay achieves 41% overall categorical accuracy and a thermal-label Peirce skill score of +0.107 (see `docs/findings/2026-06-12-historical-baseline.md` and `docs/2026-06-13-corrected-methodology-rework.md` for the headline numbers). The Phase C spike is a deliberate test: can a learned model improve materially on +0.107, or is the heuristic near the data ceiling for the available features? ## 2. Physics foundation (cite, don't re-derive)

The feature set for the ML model inherits the domain predictors from the rule baseline. The thermal engine is **slope heating on the Herzogstand/Jochberg south-facing flanks**, drawing cool air from the Kochelsee basin through a Düseneffekt funnel and producing a N-to-NNE thermal wind across the lake — the so-called "Thermik" or "Walchensee-Thermik" [36]. The Thermik is suppressed or destroyed by Föhn (Bolzano−Innsbruck pressure delta ≥ 4 hPa), by 150–210° upper-level flow, by low cloud over the lake, by post-rain wet soil, and by extended heatwaves that shrink the day/night temperature differential [36]. The current rule set encodes each of these as a discrete signal (NO_GO / MAYBE / GO) and aggregates them in a severity-tiered scheme where a single hard NO_GO wins outright and two or more soft NO_GOs downgrade to MAYBE [38, 36]. The 15–20 candidate features for the ML spike are documented in `docs/future-factors.md` [37]: dew-point spread, boundary-layer height, soil moisture + 24h precipitation, CAPE + Lifted Index, low-cloud cover, multi-level wind shear, MSL pressure differential (Munich−Innsbruck and Innsbruck−Bolzano), and the standard Open-Meteo suite (solar radiation, 850 hPa wind, 700 hPa wind, etc.). The reader who needs the exact definitions of these signals, their thresholds, and their physical motivation should consult those two in-repo documents; this report treats them as a fixed input space.

The one important domain-specific point that the rest of the report relies on: the in-season ground-truth label distribution is **substantially imbalanced** (NO_GO is roughly half the days, MAYBE roughly a third, GO the rest), so accuracy is the wrong objective and the choice of categorical skill score is not academic. The rule baseline's current +0.107 Peirce score is the bar the ML model has to beat.

## 3. Part 4 — Small-n tabular ML methodology

### 3.1 Model families — GBDTs win, deep nets lose, TabPFN is the wild card

The single most important empirical finding for this project is that **tree-based models remain state-of-the-art on medium-sized tabular data**, and that "medium-sized" is a wide bucket that comfortably contains 1–2k rows. Grinsztajn, Oyallon & Varoquaux (NeurIPS 2022) benchmarked tree-based and deep tabular models on 45 datasets, with a 20,000-compute-hour hyperparameter search per learner, and concluded that "tree-based models remain state-of-the-art on medium-sized data (~10K samples)" — the relevant inference being that below 10k the tree advantage is at least as large, because the deep models' edge comes from regularisation tricks that pay off when there's enough data to learn from [1]. Three reasons given for the gap: trees are **robust to uninformative features** (a tree simply never splits on a noise feature, while a neural net's weights move on it), trees **handle irregular target functions** natively (the lake-thermal response is sharp and threshold-like — "solar above 600 W/m² AND dew-spread above 5°C AND no foehn" is exactly the kind of decision surface a tree learns cleanly), and trees **preserve data orientation** along the original feature axes (no rotation) [1]. Shwartz-Ziv & Armon (Information Fusion 2022) corroborate with a separate benchmark: XGBoost outperformed the specialised deep models on the very datasets those deep models were originally proposed on, and required much less hyperparameter tuning [2]. An ensemble of XGBoost + deep model can beat XGBoost alone, but XGBoost remains the strongest single family [2].

For the project's specific 1,900-row regime, **the safest baseline is `sklearn.ensemble.HistGradientBoostingClassifier`** [16]. It is the modern, histogram-binned successor to `GradientBoostingClassifier` and supports: (a) `class_weight='balanced'` and a per-class weight dict, both of which are needed for the imbalanced GO/MAYBE/NO_GO target; (b) native NaN handling — at each split the tree grower learns whether missing-value samples go left or right based on potential gain, so block-missing features (e.g. the ICON-only predictors that are NaN pre-2023) need no imputation; and (c) `min_samples_leaf` tuning, which is the only effective regularisation lever at small n [16]. The implementation bins features into at most 255 integer bins (plus one reserved for NaN), which both speeds training and provides implicit regularisation [16].

**Caveat on the speed advantage.** The sklearn docs are explicit that "this estimator is much faster than `GradientBoostingClassifier` for big datasets (`n_samples >= 10 000`)" [16]. The project is well below that threshold; the speed argument for choosing `HistGradientBoostingClassifier` over `GradientBoostingClassifier` is therefore not "it's fast" but "it has `class_weight`, native NaN handling, and the binned-feature regularisation." Training time on 1,900 rows is a non-issue for either implementation.

**TabPFN is the active threat.** The "trees beat deep nets" finding is not the final word at this sample size. McElfresh et al. (NeurIPS 2023) showed that **for very small training sets (≤ 3,000 samples), TabPFN — a transformer that performs in-context learning on tabular data — outperforms all other tested algorithms** (gradient boosting, random forests, deep tabular nets) across a range of benchmarks [4]. The mechanism is that TabPFN is pre-trained on millions of synthetic tabular tasks and then conditions on the labelled examples it sees at inference time, so it effectively has a Bayesian prior over tabular prediction problems. The follow-up TabPFNv2 (Hollmann et al., Nature 2025) extends this to up to 10,000 samples and adds ensembling [4]. The project sits squarely in TabPFN's sweet spot (1,900 in-season rows, < 3,000). **The Phase C benchmark must include TabPFN as a third family** alongside GBDTs and a small deep tabular net (e.g. FT-Transformer or a tuned MLP), not as an afterthought. If TabPFN wins, the "what model?" question for production is a one-liner; if it loses to GBDT, the project has a clean story for the rule-baseline comparison.

**What about deep nets and FT-Transformer?** The Grinsztajn and Shwartz-Ziv results suggest that a plain FT-Transformer / TabNet / SAINT will lose to GBDT here, but the "ensemble of deep + GBDT beats GBDT" result from Shwartz-Ziv & Armon [2] implies a stacking ensemble (e.g. GBDT predictions + deep-net predictions → logistic meta-learner) is worth a separate ablation. This is cheap to test once the GBDT and TabPFN baselines exist.

### 3.2 Probability calibration — temperature scaling wins for multiclass

The oracle's downstream use is **probabilistic**: a rider wants to know "what's the probability the lake will fire this afternoon" not just "is the verdict GO/MAYBE/NO_GO." The historical baseline categorises from a probability, but the dashboard and the (eventual) ensemble-forecast layer want calibrated class probabilities. Three calibrators are available in sklearn; their trade-offs matter.

**Sigmoid (Platt scaling).** A one-parameter logistic regression on the classifier's logits. Strengths: works well on small calibration samples (the docs say it is "most effective for small sample sizes or when the un-calibrated model is under-confident" [3]), has an intercept that shifts decision boundaries (useful when the underlying classifier is biased toward the majority class — the GO/MAYBE/NO_GO target is exactly this) [3]. Weakness: assumes the calibration error is symmetric and normally distributed with equal variance per class, which is violated under heavy class imbalance [3].

**Isotonic regression.** Non-parametric step-function calibrator. Strengths: more flexible, performs as well or better than sigmoid when there are more than ~1,000 calibration samples. **Weakness: "isotonic calibration is not recommended when the number of calibration samples is too low (≪ 1000) since it then tends to overfit"** [3]. The project's 1,900 rows — split between train, calibration, and test — leaves the calibration set below 1,000 in most splits. Isotonic is therefore a no-go for the initial Phase C run.

**Temperature scaling (Platt's generalisation to multiclass).** A single scalar τ is fit so that the classifier's log-probabilities, divided by τ and re-softmaxed, minimise negative log-likelihood on a held-out calibration set. Temperature scaling is **now natively supported in `CalibratedClassifierCV` as of sklearn 1.8** (the `method='temperature'` option) [3]. It is the right default for the project's 3-class output: it is one parameter (no overfit risk on small calibration sets), it preserves the model's argmax (the predicted class is unchanged unless the calibrator is told otherwise), and it works on the full probability vector rather than per-class OvR.

**Why not sigmoid-OvR for multiclass?** The sklearn multiclass-calibration example is explicit: "All in all, the One-vs-Rest multiclass-calibration strategy implemented in `CalibratedClassifierCV` should not be trusted blindly" [3]. The reason is that each binary calibrator sees only the `rest` class, fits a sigmoid on it, and re-normalises — and the resulting mapping can cross class-assignment boundaries and silently change the predicted class after calibration. The same example notes that on its 1,000-row dataset calibration improved log-loss substantially (1.327 → 0.549) but the Brier score did not improve, illustrating that calibration benefit is metric-dependent [3].

**Recommendation for Phase C:** use `method='temperature'` with the calibrator fit on a held-out calibration set (not on training data — "using the classifier output of training data to fit the calibrator would thus result in a biased calibrator that maps to probabilities closer to 0 and 1 than it should" [3]). A clean split: train on ~1,400 rows, calibrate on ~250, test on ~250 (year-blocked, see §3.6). This is the right place to use `CalibratedClassifierCV(cv='prefit')` rather than the default k-fold, because the year-blocked holdout is the calibration set by construction.

### 3.3 Ordinal vs multinomial targets

The current rule baseline returns an ordinal label — GO > MAYBE > NO_GO — and a skilled rider would weight a misclassification by the gap: a NO_GO that turned out GO is a wasted day; a GO that turned out NO_GO is a missed day. The two errors are not symmetric, and a probabilistic model that returns a 3-class probability vector can be aggregated differently depending on how the cost is encoded.

**Multinomial classification** treats GO/MAYBE/NO_GO as unordered, fits one set of logits, applies softmax, and is what `LogisticRegression`, `HistGradientBoostingClassifier`, and essentially every standard sklearn classifier do. Class imbalance is handled by `class_weight='balanced'`, per-class weights, or post-hoc thresholding.

**Ordinal regression** (a.k.a. ordinal classification, proportional-odds logistic regression) parameterises the cumulative probabilities Pr(y ≤ k | x) = σ(θ_k − w · x) and fits a single coefficient vector w plus K − 1 thresholds θ_1, …, θ_{K−1} [5]. The thresholds are constrained to be ordered, which is the "proportional odds" assumption. The motivation is exactly the project's situation: the model has only one weight vector to learn per feature but is told that the distance between GO and MAYBE is the same as between MAYBE and NO_GO, which is a strong inductive prior.

**Caveat for small n.** McCullagh (1980) [11] gives the canonical formulation, but the proportional-odds assumption can be too restrictive at 1,900 rows: if the GO-vs-MAYBE decision boundary is feature-shaped differently from the MAYBE-vs-NO_GO decision boundary, forcing a single w will underfit. The Wikipedia summary is accurate on the theory [5] but the practical recommendation for a 3-class target at small n is **multinomial classification with cost-sensitive class weights as the primary baseline, and ordinal regression (a cumulative-link or `mord.LogisticAT` model) as a secondary ablation**. If the ordinal model's validation log-likelihood is not substantially better than the cost-sensitive multinomial's, prefer the latter — the cost matrix is an explicit, auditable knob, while the proportional-odds constraint is implicit and hidden.

**Practical note for the heuristic baseline's continuity.** The current rule baseline aggregates with severity-tiered semantics (a single hard NO_GO wins; two or more soft NO_GOs downgrade to MAYBE; otherwise GO) [38]. The ML model doesn't inherit this. It outputs a probability vector; the same C/L decision rule from §4.4 picks the operating point. This is a feature, not a regression: the ML model's calibration lets the user pick the threshold for their C/L ratio, which the current heuristic's discrete output does not.

### 3.4 Cost-sensitive learning

The motivating asymmetry for the Walchensee classifier is well established in the brief: **a missed session (forecast NO_GO, lake fires) costs the rider roughly 2x a wasted drive (forecast GO, lake doesn't fire)** — the rough rule of thumb from the planning notes. Cost-sensitive learning encodes this directly. Elkan (2001) [12] gives the canonical reference; the Wikipedia summary of the field is accurate [6] and sufficient for engineering context, but the primary reference should be cited. Two practical ways to encode the cost matrix:

- **Per-class `class_weight` dict.** The most direct path: pass `class_weight={'go': 2.0, 'maybe': 1.0, 'no_go': 1.0}` to `HistGradientBoostingClassifier.fit` (the `class_weight` parameter, available since v1.2, multiplies the per-sample weight and the `balanced` mode's inverse-frequency weight, so they compose cleanly [16, 17]). The risk: this is a single-number reweighting, not a full cost-matrix loss; it can shift decision boundaries but not guarantee cost-optimal decisions at all thresholds.
- **Post-hoc threshold tuning on the cost matrix.** Fit the model to minimise log-loss (probability-correct, no cost baked in), then sweep the probability→decision threshold to minimise expected cost on a held-out calibration set. This decouples model quality from operating-point choice and is the standard approach in production classifiers. Wikipedia notes the practical limitation that "the cost matrix itself is hard to specify reliably and may evolve over time" [6] — the project's 2x weight is itself a guess and should be treated as a knob, not a constant.

**A subtle point about `balanced` mode.** The `'balanced'` option in `compute_class_weight` sets each class weight to `n_samples / (n_classes * np.bincount(y))` [17], an inverse-frequency scheme with a 20-year pedigree in the rare-events literature (King & Zeng 2001, "Logistic Regression in Rare Events Data" [13] — note that the sklearn docs typo the author name as "King, Zen" and this is worth correcting if the citation is reproduced in code comments). The balanced heuristic is a reasonable starting point but it does **not** encode the missed-session vs wasted-drive asymmetry; it only corrects the prior. For Phase C, the right move is to start with `class_weight='balanced'` and then explore the 2x reweighting on the GO class as a separate ablation.

### 3.5 Imbalance handling

The 3-class target is substantially imbalanced. NO_GO is roughly half the days, MAYBE roughly a third, and GO the rest (rough estimate from the historical replay; exact distribution to be confirmed from `data/replay_full.csv` before the spike). Three families of approaches are available, in increasing order of intervention:

- **No intervention (raw counts).** Works if the model family has a log-loss-based objective (gradient boosting with cross-entropy, logistic regression, TabPFN's NLL training). The model learns the prior; the threshold is tuned post-hoc. This is the cleanest baseline.
- **Reweighting (`class_weight='balanced'` or a custom dict).** Adjusts the loss so each class contributes proportionally to its weight, not its count. The standard go-to for sklearn classifiers.
- **Resampling (oversample minority, undersample majority, SMOTE).** Modifies the training data so the class distribution is uniform. Risky on small n: SMOTE creates synthetic samples that can leak across the temporal autocorrelation, and undersampling throws away the majority class's information.

**For Phase C, the recommendation is: start with no intervention (raw counts) + post-hoc threshold tuning; if the 3-class confusion is dominated by NO_GO→MAYBE confusion, add `class_weight='balanced'`; if cost-sensitive weights are needed, switch to the per-class dict.** Resampling should be reserved as a last resort and used only with year-blocked folds to prevent synthetic samples from one year contaminating another. The general warning in the imbalance literature — that resampling can interact badly with temporal autocorrelation — applies especially to SMOTE on autocorrelated daily data.

### 3.6 Leakage-aware temporal validation

This is the **most important methodological decision for the project** and the one most likely to be done wrong by default. The default sklearn validation is `KFold(5)`, which shuffles all 1,900 rows into five folds and trains on 80% / tests on 20% — but because the rows are **consecutive days of weather data with strong temporal autocorrelation** (yesterday's pressure pattern is correlated with today's), this setup trains on the future to predict the past, which produces optimistically biased scores and gives no honest estimate of the model's performance on a yet-to-be-seen season. The same critique, formalised, is the subject of Roberts et al. (2017) in *Ecography* [9] and the *WMO/WWRP/WGNE Joint Working Group on Forecast Verification Research* guidance [20]; the sklearn `common_pitfalls` page demonstrates the failure mode empirically by showing that fitting feature selection on all data before splitting produces 0.76 accuracy on random noise features, where a correctly-pipelined fit gives 0.5 [15].

**Two correct validation strategies, both needed:**

- **`sklearn.model_selection.TimeSeriesSplit`** provides forward-chaining train/test indices: in the k-th split the first k folds are train and the (k+1)-th is test [12]. Successive training sets are nested supersets, not independent partitions. This is the right tool for **hyperparameter selection** and **within-development comparison** because it uses the data efficiently (every row is a test row exactly once).
- **A year-blocked holdout.** For the **final claim** that "the model achieves X Peirce score on 2023+ data, beating the rule baseline's Y," the honest test is to train on years ≤ 2022 and test on years ≥ 2023. This is a one-shot evaluation that mirrors the production deployment scenario (the model trained on the past, asked to predict a future it has not seen). The ICON-vs-IFS era split visible in the historical baseline findings (`docs/findings/2026-06-12-historical-baseline.md`) makes the year-blocked holdout even more important: the model is being asked to generalise across a data-source change, which is exactly the kind of extrapolation Roberts et al. recommend testing for. The paper's broader recommendation — paraphrased rather than quoted verbatim, see §6 — is that **cross-validation folds should be separated to match the intended prediction scope**, and that spatial/temporal/hierarchical blocks are the standard way to do this. The training and test data should be structured to test whether the model performs as well on more distant (in time) data as on nearby, and failure to do so signals unaccounted structure, overfitting, or missing predictors. The original "spatial and environmental blocks" sentence that gets quoted from Roberts et al. is, on closer reading of the source, a paraphrase of a recommendation in the paper's discussion of the `blockCV` family of methods; the exact wording in the paper itself describes "split[ting] data into 'blocks' at some central point(s) of the dependence structure, such as in time or space" [9]. For the year-blocked holdout, train on years 2017–2022 and test on years 2023+; for robustness, also test on a contiguous subset of years (e.g. train 2017–2021, test 2022) as a stress test for the 2021/2022 anomaly years [38].

**Data leakage, the sklearn way.** The `common_pitfalls` page is the cleanest practitioner reference: data leakage "occurs when information that would not be available at prediction time is used when building the model" and produces "overly optimistic performance estimates, for example from cross-validation, and thus poorer performance when the model is used on actually novel data" [15]. The fix is mechanical: any preprocessing (standardisation, imputation, feature selection, even target encoding) must be `.fit()` only on the training fold, then `.transform()` on the test fold. The sklearn page demonstrates this with a 0.76 vs 0.5 accuracy example on random noise [15]. For the project, this means: `fit_transform` only on the train rows, never on the combined train+test rows; if using `SimpleImputer` for the block-missing features, fit it on train only; if using SHAP-based feature selection (a natural Phase C experiment given the rule-based prior), the SHAP values are computed on the model trained on the train fold only.

**Reproducibility.** The `common_pitfalls` page also notes that "for reproducible results across executions, remove any use of `random_state=None`" and that passing integers to CV splitters is the safest option [15]. Phase C should set `random_state` consistently on every stochastic component (`HistGradientBoostingClassifier`, `TimeSeriesSplit`, `train_test_split`, `TabPFN`'s internal randomness if exposed). Without this, the year-blocked holdout scores will jitter on re-runs and small claimed improvements won't be reproducible.

### 3.7 Feature selection & overfit avoidance

At 15–20 features and 1,900 rows, the per-feature sample count is 95–127, which is not overfitting-impossible but is also not the regime where feature selection matters most. Three layers of defence:

- **Model-intrinsic feature selection.** Tree-based models never split on uninformative features (a non-split feature contributes nothing to the loss) [1]. This is the strongest argument for GBDT over deep nets at this sample size: the model can be handed all 20 features including weakly-predictive ones and the irrelevant ones will be silently ignored. Phase C should start with the full feature set and let the tree rank features by importance / SHAP.
- **SHAP-based feature selection (secondary).** A separate ablation: fit the model on the full feature set, compute SHAP values, drop features with near-zero mean |SHAP| across a held-out fold, refit. This is a valid pipeline as long as the SHAP computation is part of the training fold and is never informed by the test fold.
- **Manual selection from the rule baseline.** The 14-rule baseline is effectively a human-engineered feature subset (each rule consumes 1–2 features). Training the ML model on the same 14 features and comparing to the full 20-feature model is a meaningful ablation because it isolates the value of learned interactions from the value of feature engineering.

The overfit levers to keep in mind for `HistGradientBoostingClassifier`: `min_samples_leaf` (raise for smoother decision surfaces; lower for finer splits), `max_depth` (cap at 4–6 for 1,900 rows; deeper trees overfit), `l2_regularization` (default 0 is fine for a first pass; raise if the train/test gap on the year-blocked holdout is large), and `max_iter` (more iterations → more overfit; use `validation_fraction` + `n_iter_no_change` to early-stop on a held-out validation fold) [16]. For TabPFN, the only knob is the in-context example count; for deep tabular nets, weight decay and dropout are the primary regularisers.

### 3.8 Block-missing features (NaN handling)

The replay data has a known block-missing structure: the ICON-era predictors (anything computed from `data/replay_*.csv` after the 2022-11-24 ICON archive cutover) are NaN for pre-2023 days. This is exactly the regime `HistGradientBoostingClassifier` is designed for — its docs say "this estimator has native support for missing values (NaNs). During training, the tree grower learns at each split point whether samples with missing values should go to the left or right child, based on the potential gain" [16]. The implication: the model can be trained on the union of the IFS-era and ICON-era data without imputation, and the learned split policy encodes "for the years 2017–2022 this signal isn't measured, so use the other features instead." This is a strong inductive bias — better than mean-imputation (which would tell the model the IFS-era has "average" soil moisture when in fact it has unmeasured soil moisture) and better than dropping the rows.

**Caveat for the year-blocked holdout.** If the model is trained on IFS-era data and tested on ICON-era data, the ICON-era test rows have measured values for the ICON predictors and NaN for the IFS-only predictors. The native NaN handling routes these correctly, but the model has never seen an ICON-era row during training, so the routing is operating on prior probabilities of "what does an ICON row look like" inferred from the IFS-era training data. This is the right test for the rule-baseline comparison — the model is being asked to do exactly what the production deployment will do — but the model should be allowed to fall back to features that are measured in both eras. A feature-importance plot at the end of Phase C is the cleanest way to confirm the model is using the ICON-stable features (pressure delta, solar radiation, dew-spread, BLH) rather than relying on the ICON-only signals it has never seen.

A second layer: features that are block-missing because of upstream data-source changes (like the ICON/IFS cutover) should be **flagged in the feature metadata** so that future schema changes can be detected. The current replay script captures the model source per day, and that should be carried through into the ML feature matrix as an `era` indicator that the model can optionally consume (e.g. as a `sample_weight` weight or an additional binary feature). Whether to feed the era indicator into the model is a design decision: including it lets the model learn different decision surfaces for the two eras, which may or may not be what production needs; excluding it forces the model to use only features that generalise across the era boundary, which is the harder but more honest task.

## 4. Part 5 — Forecast verification metrics

The rule baseline's +0.107 Peirce score is the bar. The ML model has to beat it on the same scoring protocol, on the same test set, with a statistically-significant difference. The following subsections establish which metrics, which significance tests, and which traps.

### 4.1 Why accuracy is wrong

The standard "fraction correct" accuracy metric is the wrong objective for the GO/MAYBE/NO_GO target, for two reasons that are both well-documented in the literature.

First, **class imbalance makes accuracy dominated by the majority class**. A trivial classifier that always predicts NO_GO would, on the project's roughly 50% NO_GO distribution, score ~50% accuracy despite being useless [7]. The historical baseline finding (41% overall categorical accuracy on a 3,331-day replay) is itself partially a function of this dominance; the more diagnostic number is the 6.3% hard-error rate (days the rules said go when the lake didn't fire, or vice versa) [38].

Second, **accuracy doesn't decompose into the components a forecaster can act on**. A Peirce score, in contrast, decomposes into hit rate minus false alarm rate and tells you whether the model is failing on the rare class or the common class; a Brier score decomposes into reliability, resolution, and uncertainty and tells you whether the failure is calibration, discrimination, or inherent noise (see §4.3). Accuracy gives you one number with no information about which lever to pull.

The WMO/WWRP/WGNE Joint Working Group on Forecast Verification Research FAQ is explicit on this: "can be misleading since it is heavily influenced by the most common category, usually 'no event' in the case of rare weather" [18]. Precision, recall, and F1 are the immediate upgrades for imbalanced binary problems, but for a 3-class ordinal target with an asymmetric cost matrix, none of them is the right final answer either — the right final answer is a skill score that zeros out constant or random forecasts and is interpretable as "improvement over a reference forecast."

### 4.2 Categorical skill scores

The categorical scores that matter for this project, in order of utility:

**Peirce Skill Score (PSS) / Hanssen-Kuipers Discriminant (HKD) / True Skill Statistic (TSS).** Mathematically `J = sensitivity + specificity − 1 = TP/(TP+FN) + TN/(TN+FP) − 1` [14], so it's the difference between the hit rate and the false alarm rate, and the equivalence between Youden's J, Peirce's skill score, and the Hanssen-Kuipers discriminant is established in the diagnostic-test literature (Youden 1950 [14]). The WMO/WWRP FAQ describes it as "does not depend on climatological event frequency" [18], which is the key property for the project's imbalanced target — a constant forecast scores 0, a perfect forecast scores 1, an inverted forecast scores −1. The current rule baseline's +0.107 means the rules are barely above chance on the binary thermal-vs-not-thermal classification; the ML spike needs to land on the other side of 0.3 to be a meaningful improvement.

A subtle point worth flagging: the WMO/WWRP page also notes that for rare events the Peirce/HK discriminant can become "unduly weighted toward the first term (same as POD)" and may be "more useful for more frequent events" [18]. The formula is independent of climatological frequency in the algebraic sense, but the score's discriminative behaviour degrades when the event is rare enough that small absolute changes in the hit rate dominate the score. The project's GO fraction is not rare in the meteorological-rare-event sense (it's roughly 20% of in-season days), so this caveat is a hedge rather than a deal-breaker — but if the project ever moved to a finer-grained target (e.g. "wind above 20 kt" rather than "lake fires"), the rare-event caveat would become load-bearing.

**Heidke Skill Score (HSS).** `HSS = (correct − expected_correct_by_chance) / (total − expected_correct_by_chance)`, ranging from −∞ to 1, with 0 indicating no skill and 1 perfect [18]. The CAWCR FAQ says it "measures the fraction of correct forecasts after eliminating those forecasts which would be correct due purely to random chance" [18]. HSS is a **generalisation to multi-category** of the more common 2x2 form: for K > 2 it requires defining a chance table. For the project's 3-class target, HSS is the right multi-category skill score — but it doesn't disentangle the per-class performance the way Peirce does. Use it as a complement to Peirce on the binarised thermal/no-thermal target, not as a replacement.

**Gerrity score.** A multi-category skill score that assigns non-uniform "distance" penalties to off-diagonal cells of the contingency table. For the project's ordinal target, the Gerrity score is the natural multi-category upgrade of HSS because it encodes that GO→NO_GO is a worse error than GO→MAYBE. **The brief mentions it but the aggregated scout output did not produce a primary reference for the Gerrity score**; the canonical references are Gandin & Murphy (1992, *Mon. Wea. Rev.* 120:2709-2717) and the Jolliffe & Stephenson textbook *Forecast Verification: A Practitioner's Guide in Atmospheric Science* [25]. Phase C should consider the Gerrity score as a secondary metric and weight its cost matrix by the missed-session vs wasted-drive asymmetry from §3.4.

**Brier skill score family.** A generalisation of Peirce-style categorical skill to the probabilistic case, defined as `1 − BS_forecast / BS_reference` where the reference is typically a climatological forecast. For a binary event, BSS = 1 at perfect, 0 at climatology, and negative for worse-than-climatology. The 3-class ordinal target needs the **ranked probability score** version, covered in §4.3.

**Bottom line for the categorical scoring protocol:** the primary metric is the Peirce score on the binarised thermal/no-thermal target (matches the rule baseline's existing +0.107 anchor); the secondary metric is the multi-category Gerrity score weighted by the cost asymmetry; the tertiary is the 3-class accuracy and the hard-error rate (for the operational dashboard). HSS is reported for completeness but not used as the primary decision criterion.

### 4.3 Probabilistic scores

For a probabilistic forecast, the model returns a 3-vector of probabilities and the metric has to be proper (incentive-compatible — i.e. the optimal forecast under the metric is to report the true probability) and ideally decomposable (so the forecaster can attribute failure to calibration vs discrimination).

**Brier score (BS).** The mean squared error of the predicted probability vector: `BS = (1/N) Σ (p_i − o_i)²` where p_i is the forecast probability and o_i is the 0/1 outcome [9]. Brier (1950) [9] is the original reference; Murphy (1973) [10] gave the decomposition `BS = REL − RES + UNC` — reliability (calibration error, 0 for perfectly calibrated), resolution (discrimination, positive for better-than-climatology), and uncertainty (inherent entropy of the climatology). **The sign convention is REL = 0 for perfect calibration, RES = positive (higher is better discrimination), UNC = inherent uncertainty of the climatology** [10]. A good forecast has low REL, high RES, and a BS below the climatological reference. The Murphy decomposition is the most actionable diagnostic in the whole toolkit because it tells you whether to spend engineering effort on calibration (high REL) or feature engineering (low RES).

**Brier sensitivity to climatology.** The WMO/WWRP FAQ and Benedetti (2010, *Mon. Wea. Rev.* 138:203-211) both note that the Brier score is sensitive to the climatological frequency of the event — for a rare event a "good" BS is achievable simply by predicting the climatological probability for every day, which makes the Brier skill score (BSS) easy to game at low base rates. The Wikipedia Brier-score article is accurate on this [8] but the primary references are Brier (1950) and Murphy (1973). The project's GO base rate (~20%) is not rare in this sense, so the BSS will not be artificially inflated by a climatological forecast — but the decomposition is still the right diagnostic.

**Brier for ordinal targets: a known limitation.** The Wikipedia article is explicit: "the Brier score is appropriate for binary and categorical outcomes that can be structured as true or false, but it is inappropriate for ordinal variables which can take on three or more values" [8]. For the project's 3-class ordinal target, the **ranked probability score (RPS)** is the correct probabilistic metric.

**Ranked Probability Score (RPS).** The WMO/WWRP FAQ describes it as measuring "the sum of squared differences in cumulative probability space for a multi-category probabilistic forecast" [18]. For an ordinal target with K categories, RPS = (1/N) Σ_n Σ_k (CDF_forecast(k) − CDF_observed(k))² summed over the categories; ranges 0 (perfect) to 1 (worst). The RPS is the probabilistic counterpart to the Gerrity score: both encode the cost of being off in cumulative / ordered sense. For Phase C, RPS is the right primary probabilistic metric. The binarised Brier score on the thermal/no-thermal target is a useful complement but should be reported alongside RPS, not in place of it.

**Log-loss / cross-entropy.** Equivalent to the Brier score for binary targets up to a constant, but the more common loss function in production classifiers. Useful as a training-time diagnostic (the model's convergence is judged on log-loss) but should be reported on a held-out set alongside the proper scoring rules. Note from the sklearn multiclass calibration example: log-loss improved substantially after calibration (1.327 → 0.549) while the Brier score did not [3], illustrating that log-loss and Brier can disagree on the calibration benefit.

**Continuous Ranked Probability Score (CRPS).** The proper scoring rule for continuous probabilistic forecasts. Not directly applicable to the categorical 3-class target, but if the project ever moves to predicting the wind speed (rather than the discrete verdict), CRPS becomes the right metric. Worth keeping in mind for the future buoy-signal work documented in `docs/future-buoy-signals.md`.

### 4.4 Cost-loss economic value & relative value

The cost-loss decision model is the bridge from probabilistic forecast quality to rider value. The Wikipedia summary is sufficient background [15]: the user takes the protective action (drives to the lake) when the forecast probability of the event exceeds the cost/loss ratio C/L, where C is the cost of the action (drive, time, equipment wear) and L is the loss averted (a missed session). The expected cost of acting is C; the expected cost of not acting is p · L.

The **CAWCR relative-value framework** [22] is the operational standard. It uses a 2x2 contingency table (hits, false alarms, misses, correct negatives) combined with a cost-loss matrix (C for acting, L for not acting) to compute the economic value V of a forecast as a function of the user's C/L ratio. Three properties of the relative-value framework are critical for Phase C:

- **Maximum value is achieved where C/L equals the climatological probability** of the event. At that point, the value equals the Hanssen-Kuipers discriminant (Peirce score) [22]. This is a deep result: a perfect-discrimination forecast is worth exactly its Peirce score in relative economic value to a user whose C/L ratio matches the climatology.
- **Value is user-dependent.** Different users have different C/L ratios; a single forecast has different utility to each. Verification by economic value must sweep across the C/L spectrum and report the value curve, not a single number [22].
- **Probabilistic forecasts only achieve optimal value when they are calibrated.** A miscalibrated probability estimate leads the user to take the action at the wrong C/L threshold, degrading the realised value below the forecast's potential value [22]. This is the formal justification for §3.2's calibration requirement.

Richardson (2000, *Quart. J. R. Meteorol. Soc.* 126:649-667) [27] is the seminal operational reference for relative economic value, and is the source most frequently cited for the framework. The aggregated scout output did not extract specific results from the paper (the abstract is paywalled), so the citation should be used as a primary reference for the framework's origin and motivation rather than as a direct source for any specific claim in this report. The CAWCR page [22] is the better primary source for the formulae and the worked example. **For the relative-value plot in the Phase C dashboard, the CAWCR page is the operational reference; the Richardson (2000) citation is the historical/credit reference.**

For the project, the practical recommendations are:

- **Define C/L as a parameter, not a constant.** The "missed session is 2x a wasted drive" rule of thumb is a guess. The Phase C dashboard should plot relative value across C/L ∈ [0.05, 0.95] in 0.05 steps and let the rider pick their ratio.
- **Report the value curve, not a single number.** A user with C/L = 0.1 (very cost-sensitive, only drives on near-certainties) sees a different value from a user with C/L = 0.5 (drives whenever the forecast is coin-flip). The single-number summary is the area under the value curve, which is the closest analogue to a single "utility" metric.
- **Use the relative-value framework to set the operating threshold for the discrete verdict.** If the user picks C/L = 0.2, the optimal threshold is the probability p* that maximises value at C/L = 0.2, which is the threshold the model uses to map its 3-vector to a GO/MAYBE/NO_GO decision. This is the principled way to convert a probabilistic classifier to the discrete verdict the current dashboard displays.

### 4.5 McNemar's test & paired significance testing

The Phase C comparison is between two classifiers evaluated on the **same days**: the rule baseline and the ML model. The standard significance test for this exact setting is McNemar's test, operating on the 2x2 disagreement table (rule correct / ML wrong vs rule wrong / ML correct). The null hypothesis is that the two models have the same error rate; rejection means one is significantly better than the other on this test set.

Dietterich (1998, *Neural Computation* 10:1895-1923) [5] is the canonical reference. The key recommendation, paraphrased: **for algorithms that can be evaluated only once on a fixed test set, McNemar's test has acceptable Type I error and is the recommended choice; for algorithms that can be evaluated multiple times via k-fold cross-validation, the 5x2cv paired t-test (also called the Dietterich test) is the recommended choice.** The project has both modes — the year-blocked holdout is a fixed single evaluation, while the `TimeSeriesSplit` folds are multiple evaluations — and both tests are needed.

**McNemar's test mechanics.** Standard form: `χ² = (b − c)² / (b + c)` where b and c are the two disagreement cells (rule right / ML wrong, and rule wrong / ML right). The exact binomial version should be used when either disagreement cell is < 25; the χ² approximation is only valid when both cells have ≥ 25 [8, 27]. For the project, the year-blocked test set is roughly 1,257 ICON-era days (2023+); a 5% improvement in Peirce (from 0.107 to 0.157) on that test set would correspond to roughly 60 more correct predictions, which is comfortably above the χ² threshold. But the test should be run with `exact=True` to be safe, and the result should be reported with effect size (Peirce difference) and confidence interval, not just a p-value.

**5x2cv paired t-test mechanics.** For k-fold cross-validation, Dietterich's 5x2cv test takes 5 replications of 2-fold CV, computes the paired difference on each fold, and tests whether the mean difference is zero. The test has better Type I error control than the naive paired t-test on the k-fold differences. For Phase C, the 5x2cv test is the right significance test when comparing two model *families* across the `TimeSeriesSplit` folds (e.g. is `HistGradientBoostingClassifier` with max_depth=4 significantly better than `HistGradientBoostingClassifier` with max_depth=6, on average across the year-blocked folds?). The MLxtend implementation by Raschka is the standard reference implementation and exposes the test as `mcnemar.perm_test` (for the exact version) and a 5x2cv helper [27].

**Multiple-comparisons correction.** Phase C will run a hyperparameter sweep over `HistGradientBoostingClassifier` (max_depth ∈ {3, 4, 5, 6}, learning_rate ∈ {0.01, 0.05, 0.1}, min_samples_leaf ∈ {5, 10, 20, 50}) — that's 48 hyperparameter combinations. Comparing the best of 48 to the rule baseline with a single McNemar's test is invalid; the family-wise error rate balloons. **Use the closed testing / Bonferroni-Holm correction** when reporting a "best model" result, or — preferably — split the sweep into a "tuning" phase (no significance tests, just select the best by cross-validated score) and a "comparison" phase (run McNemar's on the tuned model vs the rule baseline on the year-blocked holdout, which is a single test). The tuning/comparison split is the cleanest way to avoid the multiple-comparisons trap.

**A note on effect size vs significance.** With 1,257 test days, even a 1-percentage-point improvement in Peirce will be statistically significant. The more meaningful question is whether the improvement is **operationally meaningful** — does it change a rider's decision on enough days to matter? The dashboard's value curve (§4.4) is the right tool for this: report the area under the value curve, and the C/L range over which the value is meaningfully higher than the rule baseline's. A 1% Peirce improvement that translates to 0.02 more area under the value curve is not a meaningful improvement, even if p < 0.001.

### 4.6 Equitable-vs-non-equitable score taxonomy

A useful mental model for thinking about categorical forecast scores: they differ in which "reference forecast" they zero out.

- **Equitable scores (Heidke, Gerrity, multi-category BSS) zero out a random forecast.** The HSS of 0 means "no better than a random forecast with the same marginal distribution." A constant forecast scores some value below 0 in general.
- **Non-equitable / standard-skill scores (Peirce, Gilbert, Original Kuhn) zero out a constant forecast.** The Peirce score of 0 means "no better than always predicting the same value." A random forecast scores below 0 in general.
- **Proper scoring rules (Brier, RPS, log-loss) zero out the true climatology** in the BSS sense. A perfectly-calibrated climatological forecast has BSS = 0.

For a baseline comparison, the choice of which score to use as primary is itself a methodological decision. The current rule baseline's +0.107 Peirce is "barely above chance" (Peirce = 0 means "no better than constant"); if the project reports the rule baseline's HSS as well, the value will be different (HSS = 0 means "no better than random"). A model that has Peirce = 0.3 and HSS = 0.2 is meaningfully better than the rule baseline on both axes; a model with Peirce = 0.3 and HSS = 0.05 is a model that beats the constant forecast but not the random one, which is a sign that the model is mostly encoding the prior and not learning the conditional structure.

The recommendation: report both Peirce and HSS as primary categorical scores, with Peirce as the headline (matches the rule baseline's existing anchor) and HSS as the random-forecast sanity check. If they disagree materially, the model has a problem with conditional structure that the Peirce score is hiding.

## 5. Cross-cutting recommendations for the Walchensee classifier

A concrete to-do list, tying Part 4 (modeling) to Part 5 (verification), for the developer of Phase C:

**Data and validation.**
- Train on years 2017–2022 (IFS era, ~2,074 days), test on years 2023+ (ICON era, ~1,257 days). This is the year-blocked holdout that mirrors the production deployment scenario.
- Use `sklearn.model_selection.TimeSeriesSplit` for hyperparameter selection and `TimeSeriesSplit(gap=7)` to leave a one-week gap between train and test to mitigate short-range autocorrelation.
- Fit all preprocessing, calibration, and feature-selection steps on the training fold only; never on the combined train+test. Set `random_state=42` on every stochastic component for reproducibility.

**Model family.**
- Primary baseline: `HistGradientBoostingClassifier` with `class_weight='balanced'` and `min_samples_leaf` tuned via the year-blocked holdout.
- Second family: `TabPFN` (or TabPFNv2 if the 1,900-row regime is below TabPFNv2's 10k cap). Treat as a separate model family; re-run the appropriate significance test.
- Optional third family: a tuned deep tabular net (FT-Transformer or a 2-layer MLP with dropout) for the stacking ensemble ablation. Probably not worth doing in the first pass; revisit if GBDT and TabPFN are tied.
- Set `class_weight='balanced'` first; then ablate to a custom `{'go': 2.0, ...}` dict for cost-sensitive learning. Start with no resampling; reach for SMOTE only if neither reweighting approach moves the Peirce score.

**Probability calibration.**
- Use `CalibratedClassifierCV(method='temperature', cv='prefit')` fit on a year-blocked calibration set (e.g. year 2022 as calibration, 2023+ as test). Avoid sigmoid-OvR for the 3-class target; the OvR map can cross class boundaries.
- Do not use isotonic at this sample size; the calibrator will overfit.

**Target structure.**
- Primary: multinomial classification with cost-sensitive `class_weight` dict. Compare against the 14-rule baseline using a 3-class confusion matrix.
- Secondary: ordinal regression (cumulative-link model) as an ablation. Report the per-class threshold estimates; if they are far from uniform, the proportional-odds assumption is too strong for the data.

**Scoring protocol.**
- Headline metric: Peirce score on the binarised thermal/no-thermal target. Matches the rule baseline's +0.107 anchor.
- Secondary metric: multi-category Gerrity score weighted by the cost asymmetry.
- Probabilistic metric: RPS on the 3-class target, plus Brier score on the binarised thermal target with the Murphy (1973) decomposition reported (`BS = REL − RES + UNC`).
- Tertiary metrics: HSS on the 3-class target, 3-class accuracy, hard-error rate (matches the existing dashboard panel).
- Economic value: relative-value curve across C/L ∈ [0.05, 0.95], with the area under the curve as the single-number summary.

**Significance testing.**
- For the rule-baseline vs ML model comparison on the year-blocked holdout: McNemar's test with `exact=True` (or the χ² approximation if both disagreement cells are ≥ 25). Report effect size (Peirce difference with bootstrap CI) and p-value.
- For hyperparameter comparisons within a model family: 5x2cv paired t-test (Dietterich 1998) on the `TimeSeriesSplit` folds.
- Split the hyperparameter sweep into a "tuning" phase (no significance tests) and a "comparison" phase (McNemar on the tuned model). Do not run McNemar on the best of 48 hyperparameter combinations without a multiple-comparisons correction.

**Operational deployment.**
- Carry the `era` indicator (IFS vs ICON) through into the feature matrix as a flag, but do not feed it into the model. The model should generalise across the era boundary, not depend on it.
- Confirm via SHAP feature importance that the model's decisions are driven by the ICON-stable features (pressure delta, solar, dew-spread, BLH) rather than the era-specific signals. If SHAP shows the model relying on the ICON-only features for 2023+ predictions, that's a red flag for production deployment post-2026.
- Set the operating threshold (probability → GO/MAYBE/NO_GO decision) from the relative-value curve at the user's C/L ratio, not from a fixed threshold tuned on the training set.

**Open questions for the spike.**
- Is TabPFN competitive with or better than GBDT at 1,900 rows? (McElfresh 2023 suggests yes; Grinsztajn 2022 suggests no for the >10k regime.)
- Does cost-sensitive `class_weight` give a meaningful improvement over `'balanced'`? (Expected: yes, by 1–3 Peirce points, but the 2x weight is itself a guess.)
- Does temperature scaling give a meaningful improvement over the raw `predict_proba`? (Expected: yes, on log-loss; unclear on Peirce, since Peirce is rank-based and calibration doesn't change ranks.)
- Does ordinal regression beat multinomial? (Expected: no, because the proportional-odds assumption is too strong for this feature set.)
- Does the model beat the rule baseline on the 2023+ holdout, with a statistically-significant McNemar's test and a meaningful relative-value improvement? (Unknown; this is the spike's primary outcome.)
- Does the 2021/2022 anomaly (the historical baseline's 13.4% and 18.1% hard-error years) show up in the model's residuals? (Expected: yes; the model is presumably overfit to "normal summer" conditions in the same way the rule baseline is. Adding a "weather regime" feature is a natural Phase D follow-up.)

## 6. Limitations

- **Three claims were dropped during adversarial verification.** Roberts et al. (2017) had two verbatim quotes that were not in the source paper as quoted (the actual paper uses "blocks at some central point(s) of the dependence structure" rather than the quoted "spatial and environmental blocks" sentence, and the combined "nearby vs distant places + structure/overfitting/missing predictors" quote is a paraphrase rather than a single sentence). The WMO/WWRP FAQ's Brier "sensitive to climatological frequency" verbatim quote is best attributed to Benedetti (2010) or to Murphy (1973) or paraphrased; the canonical primary sources are these peer-reviewed papers, not the FAQ page. All three concepts (block CV is the right thing for structured data; Brier is sensitive to climatology) are true and supported, but the verbatim quotes were not verifiable in the cited sources and have been replaced with paraphrases backed by the primary literature.
- **Wikipedia is accurate but secondary.** Five Wikipedia pages were used (Ordinal regression, Accuracy paradox, Cost-sensitive ML, Cost-loss model, Brier score, Youden's J) — all are accurate as of the access date but are not the primary sources. The primary sources are cited where they exist: McCullagh (1980) for ordinal regression, Elkan (2001) for cost-sensitive learning, Brier (1950) and Murphy (1973) for the Brier score, Youden (1950) for Youden's J. The Wikipedia pages are useful for the developer's reading list but should not be cited as the source of record in the Phase C paper.
- **Two sources were marked weak and downgraded.** The machinelearningmastery.com McNemar post is a blog; the primary reference is Dietterich (1998) and the McNemar 1947 paper, both of which are cited. The Richardson (2000) *Quart. J. R. Meteorol. Soc.* paper's abstract is paywalled and no specific claims were extracted from it; the paper is cited as the historical/credit reference for the relative-value framework, with the CAWCR page as the operational reference. The "published 2025-06-25" Wikipedia date for cost-sensitive learning is a last-edit date, not a publication year; the field is from Elkan (2001) and that is how it is cited.
- **The HK discriminant under rare events is hedged.** The formula is independent of climatological frequency in the algebraic sense, but the WMO/WWRP page notes that "for rare events HK is unduly weighted toward the first term (same as POD), so this score may be more useful for more frequent events" [18]. The project's GO base rate (~20% of in-season days) is not rare in this sense, so the caveat is informational rather than load-bearing, but the report hedges accordingly.
- **The 10,000-row speedup caveat for `HistGradientBoostingClassifier` is load-bearing.** The sklearn docs say "this estimator is much faster than `GradientBoostingClassifier` for big datasets (`n_samples >= 10 000`)" [16]. The project is at 1,900 rows; the speed argument for choosing `HistGradientBoostingClassifier` is its `class_weight`, native NaN, and binned-feature regularisation, not its speed. Training time is a non-issue for either implementation. `min_samples_leaf` is the more important knob at this sample size.
- **The Brier "BS = REL − RES + UNC" sign convention** follows Murphy (1973): REL = 0 for perfectly calibrated, RES = positive for better-than-climatology discrimination, UNC = inherent climatological uncertainty. The convention is sometimes written with reversed RES sign in secondary sources; the primary reference is Murphy (1973) and the sign convention used here is the standard one.
- **TabPFN benchmarks are recent (2023/2025) and the field is moving.** The Hollmann et al. (2023) NeurIPS paper and the TabPFNv2 Nature (2025) extension are the only two references; there is no meta-benchmark on the order of Grinsztajn 2022. TabPFN's "in-context learning on synthetic priors" mechanism is qualitatively different from both GBDT and deep tabular nets, and its behaviour on real-world imbalanced ordinal data has not been independently benchmarked at the scale of this project.
- **All claims are at lead-time-0.** The replay data uses the Open-Meteo archive (best-match model per day), which is the same data the production forecast engine uses for the 08:00 CET run, but does not represent the lead-time-aware inputs the production deployment will eventually need. The Phase C spike is a model-quality comparison; the operational deployment question (which forecast hours to use, how to weight the ICON-EU and ICON-D2 horizons) is a separate study.
- **The primary sources cited are not all read end-to-end.** Several primary references (Youden 1950, Brier 1950, McCullagh 1980) were cited based on the Wikipedia secondary summary and the standard reference lists in Wilks / Jolliffe & Stephenson. The developer of Phase C should read the originals before quoting specific formulae; this report cites the primary works as the source of record but does not re-derive the formulae from them.

## 8. Conclusion

The Phase C ceiling spike has a clean methodology on both sides of the modelling/verification split. On the modeling side: `HistGradientBoostingClassifier` with `class_weight='balanced'` and `min_samples_leaf` tuned is the strongest single baseline, and TabPFN is the active threat that must be benchmarked in parallel; deep tabular nets are a tertiary ablation; probability calibration should use temperature scaling on a held-out year-blocked set; cost-sensitive weights should be explored as a `class_weight` dict; ordinal regression is a secondary ablation; leakage-aware validation is year-blocked (train ≤ 2022, test ≥ 2023) with `TimeSeriesSplit` for the hyperparameter sweep. On the verification side: report Peirce (binary thermal/no-thermal) as the headline metric to match the rule baseline's +0.107 anchor, RPS (3-class) as the probabilistic metric, Gerrity weighted by cost asymmetry as the ordinal metric, and the relative-value curve across C/L ∈ [0.05, 0.95] as the economic-value dashboard panel; use McNemar's exact binomial for the year-blocked holdout comparison and 5x2cv paired t for within-family hyperparameter sweeps; separate the hyperparameter sweep from the significance test to avoid the multiple-comparisons trap. The spike's primary outcome is whether the model beats the rule baseline on the 2023+ holdout with a statistically-significant McNemar's test and a meaningful relative-value improvement; the open questions are whether TabPFN wins at 1,900 rows, whether the cost-sensitive reweighting helps, and whether the 2021/2022 weather-regime anomaly shows up in the model's residuals. If the spike is positive, the implementation plan is straightforward; if it is negative, the +0.107 Peirce score stands as the empirically-supported ceiling for the available feature set at this sample size, and the next move is feature engineering (regime indicators, neighbour pressure deltas, larger-window meteorology) rather than model architecture.
